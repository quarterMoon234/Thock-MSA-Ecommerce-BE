# 실험 설계 비교 메모

## 1. 왜 실험마다 테스트 조건이 달랐는가

이번 프로젝트의 실험들은 모두 “부하 실험”처럼 보이지만, 실제로는 검증하려는 성질이 서로 달랐다.

즉 모든 실험을:

- 같은 VU
- 같은 duration
- 같은 throughput

으로 맞추는 것이 목표가 아니라,

- **무엇을 증명하려는가**
- **그 성질을 가장 직접적으로 드러내는 입력은 무엇인가**

를 기준으로 조건을 다르게 잡았다.

핵심 원칙은 아래와 같다.

1. correctness를 검증하는 실험은 **상태 결과가 중요**
2. latency/throughput를 검증하는 실험은 **반복성과 중앙값이 중요**
3. 경쟁 구간을 검증하는 실험은 **burst 입력이 중요**
4. 메시지 backlog drain을 검증하는 실험은 **이벤트 일괄 투입이 중요**

---

## 2. Redis Lua Gate

### 검증하려는 성질

- 품절 직전 경쟁 구간에서 실패 요청을 DB 전에 잘라낼 수 있는가
- DB 락 경로 진입 수와 API 응답시간이 줄어드는가

### 왜 이런 조건을 썼는가

- 재고 `10`
- 예약 요청 `1,000건`
- `1,000 VUs`, 각 1회 요청

이 실험은 지속 부하보다 **동시 경쟁 상황**을 만드는 것이 핵심이다.

즉 Redis Lua Gate는:

- “30초 동안 계속 요청을 보냈더니”보다
- “재고 10개에 요청 1,000개가 한꺼번에 몰렸을 때”

효과가 드러난다.

### 실험 성격

- **burst contention 성능 실험**

---

## 3. Outbox

### 검증하려는 성질

- broker 장애 시 direct 발행은 이벤트 유실을 막을 수 있는가
- Outbox는 `PENDING -> SENT`로 복구되는가

### 왜 이런 조건을 썼는가

- 외부 상품 생성 요청 `2,400건`
- `50 VUs`
- broker down 상태 유지

여기서 중요한 것은 latency가 아니라:

- DB에 남은 row 수
- topic에 실제 반영된 고유 이벤트 수
- outbox 상태 수렴

이다.

즉 Outbox는 “빠른가”보다 **유실 없이 복구되는가**가 핵심이라서, 결과 집계도 HTTP latency보다 DB/outbox/topic 정합성 기준으로 잡았다.

### 실험 성격

- **장애 복구 / no-loss correctness 실험**

---

## 4. Inbox

### 검증하려는 성질

- 동일 이벤트가 여러 번 들어오면 중복 소비가 실제로 발생하는가
- Inbox가 이를 1회 처리로 제한하는가

### 왜 이런 조건을 썼는가

- 동일 재고 예약 이벤트 `100회` 중복 발행
- 재고 `100`
- 수량 `1`

여기서는 sustained load보다:

- **같은 메시지를 반복 입력했을 때 상태가 어떻게 바뀌는가**

가 중요하다.

즉 평균 응답시간이나 중앙값보다:

- 처리 수
- 중복 차단 수
- 재고 예약 증가량
- Inbox 기록 수

가 핵심 지표다.

### 실험 성격

- **duplicate consume correctness 실험**

---

## 5. Kafka Partition

### 검증하려는 성질

- 단일 파티션 직렬 소비 대비 다중 파티션 병렬 소비가 처리량을 높이는가
- 같은 주문의 이벤트 순서는 유지되는가

### 왜 이런 조건을 썼는가

- 총 이벤트 `3,000건`
- 주문 `1,500건`
- 주문당 `RESERVE`, `COMMIT` 2건
- burst 발행

여기서 중요한 것은 HTTP 요청 latency가 아니라,

- backlog를 얼마나 빨리 drain하는가
- 순서 위반이 없는가

이다.

그래서 이 실험은:

- `product-service` 내부 하네스로 동일 이벤트 세트를 burst로 발행
- single/multi consumer 구조를 비교

하는 방식으로 설계했다.

### 실험 성격

- **message backlog 처리량 / 순서 보장 실험**

---

## 6. CQRS

### 검증하려는 성질

- 장바구니 조회·생성 경로의 응답시간이 줄어드는가
- 상품 서비스 지연이 직접 전파되지 않게 되었는가

### 왜 이런 조건을 썼는가

- `20 VUs`
- 시나리오당 `30초`
- 조회 회원 `50`
- 생성 회원 `100`
- 상품 서비스 지연 주입 `900ms`

CQRS는 burst 1회보다 **지속적인 읽기/쓰기 트래픽 아래에서의 응답시간과 안정성**이 중요하다.

그래서:

- 동시 사용자 20명이 30초 동안 계속 요청
- avg, p95, 성공률, 총 요청 수 측정

구조로 설계했다.

### 실험 성격

- **sustained read/write latency + dependency isolation 실험**

---

## 7. 주문 조회 N+1

### 검증하려는 성질

- fetch join으로 실제 쿼리 수가 줄어드는가
- 그 결과 응답시간도 줄어드는가

### 왜 이런 조건을 썼는가

- 주문 `100건`
- 주문당 상품 `5개`
- `20 VUs`
- `300 iterations`

이 실험은 경쟁 구간 재현이 아니라, **읽기 경로 최적화 효과**를 보는 것이다.

그래서:

1. Hibernate Statistics로 쿼리 수 감소 검증
2. k6로 응답시간 검증

두 단계로 나눴다.

### 실험 성격

- **read path 구조 최적화 성능 실험**

---

## 8. 왜 어떤 실험은 3회 중앙값이고, 어떤 실험은 1회로 충분한가

### 3회 중앙값이 필요한 경우

- Redis Lua Gate
- CQRS
- 주문 조회 N+1
- Kafka Partition

이 실험들은:

- 응답시간
- 처리량
- 총 처리 시간

처럼 환경 변동 영향을 받는 성능 지표를 다룬다.

그래서 단일 run보다 **반복 후 중앙값**이 더 적절하다.

### 단일 run으로 충분한 경우

- Inbox

Inbox는 성능보다 상태 결과가 중요하다.

즉 핵심이:

- 100회 입력이 100회 반영되는가
- 아니면 1회만 반영되는가

이기 때문에, 기대한 correctness가 정확히 나왔는지가 더 중요하다.

### Before/After 장애 비교가 중요한 경우

- Outbox

Outbox는 성능보다 신뢰성이 핵심이라:

- direct에서 유실이 나는가
- outbox에서 유실 없이 복구되는가

를 비교하는 것이 가장 중요하다.

---

## 9. 면접에서 이렇게 설명하면 된다

> 이번 프로젝트에서 실험 조건을 일부러 통일하지는 않았습니다. 각 실험이 검증하려는 성질이 달랐기 때문입니다. Redis Lua Gate는 burst 경쟁 구간, Outbox와 Inbox는 correctness와 복구, Kafka Partition은 backlog drain과 순서 보장, CQRS와 주문 조회 N+1은 지속 트래픽 아래의 응답시간 개선을 보는 실험이었습니다. 그래서 각 주제에 맞는 입력 형태와 측정 지표를 따로 설계했습니다.

이 한 문장으로 실험 설계 기준을 설명할 수 있다.
