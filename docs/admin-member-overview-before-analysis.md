# Admin Member Overview Before Analysis

## 1. 문제 인식

현재 팀 프로젝트의 관리자 기능 관점에서는 회원의 상태를 한 번에 파악하기 어렵다.

관리자가 확인하고 싶은 정보는 분산되어 있다.

- 회원 기본 정보: `member-service`
- 최근 주문 내역: `market-service`
- 남은 예치금 또는 정산 관련 정보: `settlement-service`

즉 운영자는 여러 서비스를 오가며 정보를 확인해야 하고, 운영 목적의 단일 조회 API는 아직 없다.

## 2. 도입 배경

지금까지 팀 프로젝트의 주요 역할은 아래에 가까웠다.

- Gateway 기반 인증 중앙화
- Refresh Token 세션 제어
- 자동 배포 파이프라인 구축

이 역할들은 인증, 세션, 배포 같은 공통 기반 설계에는 강점이 있지만, 운영 목적의 실제 관리자 기능은 상대적으로 약하다.

따라서 다음 작업은 단순한 CRUD 추가가 아니라, 분산된 서비스 데이터를 관리자 관점의 단일 응답으로 통합하는 기능이 적절하다고 판단했다.

## 3. 해결하고 싶은 문제

관리자 입장에서 아래를 한 번에 볼 수 있어야 한다.

- 특정 회원의 기본 정보
- 최근 주문 요약
- 현재 남은 예치금

즉 여러 서비스에 분산된 데이터를 운영 목적에 맞는 하나의 조회 응답으로 재구성하는 것이 목표다.

## 4. 고려한 구현 방법

### 4-1. API Composition

한 서비스가 다른 서비스들의 내부 API를 호출해 응답을 조합하는 방식이다.

예:

- `member-service`가 회원 정보를 직접 조회
- `market-service`에 최근 주문 요약 요청
- `settlement-service`에 예치금 정보 요청
- 응답을 합쳐 관리자용 DTO 반환

장점:

- 구현 속도가 가장 빠르다
- 현재 구조를 크게 바꾸지 않는다
- 서비스 경계를 유지한 채 통합 조회를 만들 수 있다

단점:

- 내부 서비스 응답 지연이 전체 응답 지연으로 이어질 수 있다
- 장애 전파와 timeout 처리가 필요하다

### 4-2. CQRS / Read Model

여러 서비스 이벤트를 모아 관리자 조회 전용 read model을 별도로 구성하는 방식이다.

장점:

- 조회 응답이 빠르다
- 서비스 간 실시간 호출이 없다
- 운영자 조회 용도로 최적화된 응답 구조를 만들기 쉽다

단점:

- 구현 비용이 크다
- 이벤트 설계, 적재, 정합성 관리까지 필요하다
- 현재 목적 대비 공사가 크다

### 4-3. 별도 Admin Service

관리자용 조회만 담당하는 별도 서비스를 두는 방식이다.

장점:

- 관리자 기능이 분리되어 구조가 깔끔하다
- 관리자 기능이 커질수록 확장성이 좋다

단점:

- 서비스 운영 비용이 증가한다
- 현재 프로젝트 규모에서는 과할 수 있다

### 4-4. 서비스 DB 직접 조회

한 서비스에서 다른 서비스 DB를 직접 참조하거나, 운영 DB를 한 곳에서 조합하는 방식이다.

장점:

- 구현은 쉬워 보일 수 있다

단점:

- 서비스 경계를 깨뜨린다
- MSA 설계 의도가 흐려진다
- 이번 프로젝트 방향과 맞지 않는다

## 5. 선택한 방식

현재 요구사항과 프로젝트 규모를 고려해 **API Composition**을 선택한다.

구체적으로는:

- `member-service` 안에 관리자용 통합 조회 API를 둔다
- `member-service`가 중심 회원 정보를 보유하므로 응집도가 가장 높다
- `market-service`, `settlement-service`를 내부 API로 병렬 호출해 응답을 조합한다

## 6. 왜 이 방법이 타당한가

### 6-1. 관리자 조회는 트래픽이 낮다

관리자용 조회는 일반 사용자 메인 API보다 호출 빈도가 낮다.
따라서 현재 단계에서는 read model보다 API Composition이 비용 대비 더 적절하다.

### 6-2. 운영 목적이 분명하다

이 기능은 회원, 주문, 정산 데이터를 운영자가 한 번에 보기 위한 것이다.
즉 서비스 간 데이터를 직접 섞는 것이 아니라, 운영 목적의 단일 조회를 제공하는 것이 핵심이다.

### 6-3. 현재 팀 프로젝트 역할과 잘 맞는다

이미 인증 중앙화와 내부 인증 헤더 구조를 설계해두었기 때문에, 내부 서비스 호출에도 이 구조를 자연스럽게 활용할 수 있다.

즉 단순 조회 기능이 아니라:

- 내부 통신 신뢰 경계
- 서비스 간 조합
- 운영 목적의 조회 모델

을 함께 보여줄 수 있다.

## 7. 구현 계획

### 7-1. API 스펙

예상 엔드포인트:

- `GET /admin/members/{memberId}/overview`

예상 응답:

- 회원 기본 정보
- 최근 주문 N건 요약
- 남은 예치금

### 7-2. 구현 위치

- `member-service`

이유:

- 회원이 조회의 중심 엔티티다
- Gateway에는 비즈니스 조합 로직을 두지 않는 편이 맞다

### 7-3. 내부 호출 방식

- `market-service`: 최근 주문 요약 조회 API
- `settlement-service`: 예치금 조회 API
- 가능하면 병렬 호출
- 각 호출에 timeout 적용

### 7-4. 보안 적용

- 외부 관리자 요청은 기존 Gateway 인증 구조를 따른다
- 내부 서비스 간 호출은 기존 내부 인증 헤더를 사용한다

## 8. 구현 시 주의할 점

- 순차 호출만으로 끝내지 말고 병렬 조회를 고려할 것
- 최근 주문 N건 등 응답 범위를 제한할 것
- 관리자 API 목적에 맞는 전용 DTO를 만들 것
- 장애나 지연이 전체 응답을 무한 대기시키지 않도록 timeout을 둘 것

## 9. 기대 효과

- 관리자 입장에서 회원 상태를 한 번에 파악할 수 있다
- 분산된 서비스를 운영 목적의 단일 조회 응답으로 통합할 수 있다
- 팀 프로젝트 불릿이 공통 기반 중심에서 운영 기능까지 확장된다

## 10. 예상 이력서/포트폴리오 메시지

이 작업이 잘 마무리되면 아래 메시지로 정리할 수 있다.

- 분산된 서비스 데이터를 관리자 목적의 단일 조회 API로 통합했다
- 내부 인증 헤더 기반 서비스 호출과 병렬 조회를 적용했다
- 운영자 관점의 조회 흐름을 단순화했다

## 11. 현재 판단

이 주제는 단순 API 추가가 아니라, 팀 프로젝트의 공통 기반 역할을 유지하면서도 실제 운영 기능까지 확장할 수 있는 적절한 다음 단계다.
다만 구현은 과하지 않게 가져가야 하므로, 이번 단계에서는 read model이나 별도 admin service보다 API Composition이 더 합리적이다.
