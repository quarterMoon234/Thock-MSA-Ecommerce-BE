# Admin Member Overview After Analysis

## 1. 구현 결과 요약

관리자용 회원 통합 조회 API를 `member-service`에 추가했다.

최종적으로 관리자는 아래 정보를 한 번에 조회할 수 있다.

- 회원 기본 정보: `member-service`
- 최근 주문 요약: `market-service`
- 현재 지갑 정보: `payment-service`

엔드포인트:

- `GET /api/v1/admin/members/{memberId}/overview`

이 API는 `member-service`가 중심이 되어 다른 서비스의 내부 API를 호출하고, 응답을 하나의 관리자용 DTO로 조합하는 방식으로 구현했다.

## 2. BEFORE 대비 달라진 점

BEFORE 문서에서는 세 번째 정보 축을 `settlement-service`로 가정했지만, 구현 전에 도메인을 다시 확인한 결과 이 판단은 부정확했다.

- `payment-service`: 회원 지갑, 잔액, 예치금 성격의 정보
- `settlement-service`: 판매자 정산, 수익 배분 계산

즉 "특정 회원의 현재 상태를 운영자가 한 번에 본다"는 목적에는 `payment-service`가 더 자연스럽다.

따라서 최종 구현은 아래 3개 서비스를 조합하는 형태로 정리했다.

- `member-service`
- `market-service`
- `payment-service`

## 3. 최종 설계

### 3-1. API Composition

최종적으로 선택한 방식은 **API Composition**이다.

`member-service`가 아래 역할을 맡는다.

- 회원 기본 정보 직접 조회
- `market-service`에 최근 주문 요약 요청
- `payment-service`에 지갑 정보 요청
- 결과를 관리자 전용 응답으로 조합

즉 Gateway에는 비즈니스 조합 로직을 두지 않고, 조회의 중심 엔티티인 회원을 소유한 `member-service`가 facade 역할을 하도록 구성했다.

### 3-2. 내부 API

이번 구현에서 실제로 사용한 내부 API는 아래와 같다.

- `market-service`
  - `GET /api/v1/orders/internal/members/{memberId}/summaries?limit={n}`
- `payment-service`
  - `GET /api/v1/payments/internal/wallets/{memberId}`

두 경로 모두 `/internal/`을 포함하므로 `InternalServiceAuthFilter`의 검사를 받는다.

즉 외부 요청은 Gateway 인증을 따르고, 서비스 간 내부 호출은 `X-Internal-Auth` 헤더 기반으로 보호된다.

## 4. 구현 흐름

최종 요청 흐름은 아래와 같다.

1. 관리자가 Gateway를 거쳐 `member-service`의 관리자 API 호출
2. `member-service`에서 관리자 권한 검사
3. `member-service`에서 회원 기본 정보 조회
4. `payment-service`, `market-service` 내부 API 호출
5. 응답을 단일 DTO로 조합
6. 관리자에게 최종 응답 반환

응답에는 아래 정보를 포함한다.

- 회원 ID
- 이메일
- 이름
- 역할
- 상태
- 마지막 로그인 시각
- 지갑 잔액 / 수익 / 수정 시각
- 최근 주문 요약 목록

## 5. 순차 호출에서 병렬 호출로 개선한 이유

초기 구현은 아래처럼 순차 호출 구조였다.

1. 지갑 조회
2. 응답 대기
3. 주문 조회
4. 응답 대기
5. 응답 조합

이 구조는 구현은 단순하지만, 서로 독립적인 두 호출을 불필요하게 순서대로 기다리게 만든다.

이번 기능에서는:

- 지갑 조회
- 최근 주문 조회

가 서로 의존하지 않는다.

따라서 순차 호출보다 **동시에 시작한 뒤 결과를 합치는 방식**이 더 적절하다.

최종적으로는 `CompletableFuture`와 `TaskExecutor`를 사용해 두 내부 호출을 병렬로 시작하고, 마지막에 결과를 `join()`해서 하나의 응답으로 조합하도록 변경했다.

즉 이 개선의 목적은:

- 조회 응답 시간 단축
- 독립 호출의 불필요한 대기 제거
- 단순 API 연결이 아니라 조회 지연까지 고려한 설계라는 점을 드러내는 것

이다.

## 6. 이번 구현으로 정리한 개념

이번 작업에서 가장 헷갈리기 쉬운 부분은 아래 개념들이다.

- 동기 / 비동기
- 블로킹 / 논블로킹
- 병렬 호출
- `CompletableFuture`

이걸 현재 구현과 연결해서 정리하면 다음과 같다.

### 6-1. 동기 / 비동기

**동기**

- 호출한 쪽이 결과를 받아야 다음 흐름으로 넘어간다.
- 현재 관리자 API는 최종 응답을 한 번에 반환해야 하므로 외부 API 계약은 동기다.

즉 클라이언트 입장에서는:

- 요청
- 서버가 모든 데이터를 모음
- 하나의 응답 반환

구조다.

**비동기**

- 호출을 시작하고, 결과를 나중에 받거나 별도의 흐름에서 처리하는 방식이다.
- 예를 들면 Kafka 이벤트 처리, fire-and-forget 작업, background job이 여기에 가깝다.

이번 구현은 최종 응답을 기다려야 하므로, 전체 API를 비동기 API라고 보긴 어렵다.

### 6-2. 블로킹 / 논블로킹

이건 "기다리는 방식"의 문제다.

**블로킹 I/O**

- I/O 호출을 시작한 스레드가 결과가 올 때까지 기다린다.
- 현재 Feign 기반 내부 HTTP 호출은 기본적으로 여기에 가깝다.

즉 `payment-service`, `market-service` 호출 자체는 각각 블로킹 호출이다.

**논블로킹 I/O**

- 호출을 시작한 뒤, 결과가 아직 안 왔다고 해서 스레드가 계속 묶이지 않는다.
- 대표적으로 Netty, Reactor, WebFlux 등이 이 범주에 들어간다.

이번 관리자 조회 구현은 Feign을 사용했기 때문에 **논블로킹 I/O 구현은 아니다**.

### 6-3. 병렬 호출

병렬 호출은 "독립적인 작업을 동시에 시작하는 것"이다.

이번 구현에서:

- 지갑 조회
- 주문 조회

는 서로 의존하지 않으므로 동시에 시작할 수 있다.

즉 병렬 호출은 아래 의미다.

- 지갑 조회 먼저 끝나야 주문 조회를 시작하는 것이 아니라
- 두 호출을 동시에 시작하고
- 둘 다 끝난 뒤 결과를 합친다

### 6-4. 이번 구현은 비동기인가

엄밀히 말하면:

- 외부 API는 **동기 응답**
- 내부 호출 시작 방식은 **병렬**
- 각 내부 HTTP 호출은 **블로킹**

이다.

즉 이번 구현을 가장 정확하게 표현하면:

- **동기 응답 API 내부의 병렬 호출**
- **blocking I/O를 병렬로 fan-out 한 뒤 join 하는 구조**

정도다.

"비동기 블로킹"이라는 말은 완전히 틀렸다고 하긴 어렵지만, 실무에서는 보통 더 명확하게 아래처럼 표현하는 편이 낫다.

- 병렬 동기 처리
- 내부 병렬 호출 후 join
- fan-out / fan-in

### 6-5. fan-out / fan-in

이 표현은 이번 구현을 설명하기에 적절하다.

**fan-out**

- 하나의 요청이 여러 하위 호출로 갈라지는 것
- 이번 구현에서는 `member-service`가 두 서비스 호출로 갈라진다

**fan-in**

- 갈라진 하위 호출 결과를 다시 하나로 모으는 것
- 이번 구현에서는 지갑 정보와 주문 정보를 합쳐 최종 응답을 만든다

즉 현재 구조는:

- `member-service`가 요청을 fan-out 하고
- 결과를 fan-in 해서 관리자 응답으로 만드는 구조다

### 6-6. CompletableFuture로 병렬 호출

이번 구현에서 `CompletableFuture`는 두 가지 용도로 썼다.

- 독립적인 내부 호출을 동시에 시작
- 최종 시점에 `join()`으로 결과를 수집

중요한 점:

- `CompletableFuture`를 썼다고 해서 I/O 자체가 논블로킹이 되는 것은 아니다
- 단지 서로 다른 스레드에서 각 블로킹 호출을 동시에 진행하게 만든 것이다

즉:

- Feign 호출은 여전히 블로킹
- 하지만 두 블로킹 호출을 서로 겹치게 실행

한 것이다.

## 7. 예외 처리 설계

관리자 통합 조회는 여러 서비스를 조합하므로, 내부 서비스 하나만 실패해도 전체 응답을 정상적으로 만들 수 없다.

그래서 이번 구현에서는:

- 회원 정보 없음 -> `MEMBER_NOT_FOUND`
- 관리자 권한 없음 -> `ADMIN_FORBIDDEN`
- 내부 서비스 호출 실패 -> `SERVICE_UNAVAILABLE`

로 구분했다.

이 판단의 의미는 다음과 같다.

- 도메인 오류는 기존 의미를 유지
- 분산 조회 중 외부 의존 서비스 실패는 503으로 통일

즉 운영자도 "내 요청이 잘못된 것인지", "회원이 없는 것인지", "내부 서비스 장애인지"를 구분할 수 있다.

## 8. 테스트로 검증한 것

`AdminMemberOverviewServiceTest`에서 아래를 검증했다.

- 관리자 요청 시 통합 응답 조합 성공
- 일반 사용자는 조회 불가
- `payment-service` 호출 실패 시 `SERVICE_UNAVAILABLE`
- `market-service` 호출 실패 시 `SERVICE_UNAVAILABLE`

추가로 테스트에서는 `TaskExecutor`에 `Runnable::run`을 주입했다.

이렇게 한 이유:

- 실제 코드는 병렬 호출 구조를 유지하면서도
- 테스트에서는 비동기성 없이 결정적으로 검증할 수 있기 때문이다

즉 운영 코드는 병렬 호출, 테스트는 동기 실행으로 분리해 안정성을 높였다.

## 9. 이번 구현에서 얻은 학습 포인트

### 9-1. 서비스 조합은 "어디에 둘 것인가"가 중요하다

Gateway에 조합 로직을 넣는 대신, 중심 엔티티를 소유한 `member-service`에 facade를 두는 편이 더 응집도 높고 설명이 좋았다.

### 9-2. 내부 인증 구조는 조회 조합에도 그대로 재사용할 수 있다

기존에 설계해둔 `X-Internal-Auth` 기반 내부 인증이 이번 관리자 조회에서도 자연스럽게 재사용됐다.

즉 공통 기반 설계가 실제 기능 확장에 연결된다는 점을 확인했다.

### 9-3. 병렬화는 "비동기 그 자체"보다 "대기 시간 구조 개선"으로 이해하는 것이 맞다

이번 구현을 통해 병렬 호출이:

- 완전한 비동기 API가 아니라
- 동기 응답 API 내부의 대기 구조를 개선하는 수단

임을 더 명확하게 이해했다.

### 9-4. blocking / non-blocking은 구현 기술까지 같이 봐야 한다

`CompletableFuture`를 쓴다고 자동으로 논블로킹이 되는 것이 아니다.

현재는:

- Feign = blocking I/O
- CompletableFuture = 병렬 실행 도구

조합이다.

즉 "동시 실행"과 "논블로킹"은 같은 개념이 아니라는 점을 분리해서 이해해야 한다.

## 11. 최종 정리

이번 작업은 단순 관리자 API 추가가 아니었다.

실제로는 아래를 함께 확인한 작업이었다.

- 분산된 서비스 데이터를 운영 목적의 단일 조회 응답으로 통합할 수 있는가
- 내부 인증 헤더 기반 신뢰 경계를 유지한 채 서비스 조합이 가능한가
- 순차 호출을 병렬 호출로 개선해 응답 구조를 더 낫게 만들 수 있는가
- 동기/비동기, 블로킹/논블로킹 개념을 현재 코드와 연결해서 설명할 수 있는가

최종적으로 이번 구현은 **API Composition + 내부 인증 헤더 + 병렬 호출 + 예외 통일 처리**까지 갖춘 관리자 통합 조회 기능으로 정리할 수 있다.

## 12. 개발하며 생긴 질문 정리

### 12-1. WebClient vs FeignClient 차이는 무엇인가

둘 다 HTTP 기반 서비스 간 호출에 사용할 수 있지만, 성격이 다르다.

**FeignClient**

- 선언형 HTTP 클라이언트다
- Java 인터페이스에 HTTP 메서드와 경로를 선언하면 구현을 자동으로 만든다
- Spring Cloud 생태계와 잘 붙는다
- 서버 간 API 호출을 빠르게 붙일 때 생산성이 좋다

예:

- `@FeignClient`
- `@GetMapping`
- 인터페이스 기반 호출

**WebClient**

- 프로그래밍 방식의 HTTP 클라이언트다
- 요청 생성, 헤더, 응답 처리, 예외 처리, 재시도 등을 코드로 더 세밀하게 제어할 수 있다
- WebFlux와 함께 쓰면 논블로킹 I/O 기반 구현도 가능하다

즉 단순화하면:

- Feign = 선언형, 빠르게 붙이기 좋음
- WebClient = 제어력이 높고, 필요하면 리액티브/논블로킹 구조까지 확장 가능

이번 구현에서는 내부 API가 단순한 GET 호출이고, 인터페이스 선언만으로도 충분했기 때문에 FeignClient가 더 적절했다.

### 12-2. 백엔드 서버 간 호출은 왜 FeignClient를 사용했는가

중요한 건 "무조건 Feign을 써야 한다"가 아니다.

더 정확한 표현은:

- 이번 요구사항에는 Feign이 더 적합했다

이유는 아래와 같다.

1. 호출 대상이 명확했다

- `market-service`
- `payment-service`

각각 정해진 내부 API 한두 개만 호출하면 됐다.

2. 요청/응답 구조가 단순했다

- GET 기반 내부 조회
- 복잡한 스트리밍이나 커스텀 연결 제어가 필요하지 않았다

3. 선언형 인터페이스가 코드 가독성에 유리했다

관리자 통합 조회의 핵심은 "서비스 조합"이지 "HTTP 클라이언트 코드 자체"가 아니다.
FeignClient를 쓰면 HTTP 세부 구현보다 조합 로직에 집중하기 좋다.

4. 기존 팀 프로젝트 구조와도 맞았다

이미 `market-service`에서 Feign을 쓰고 있었고, 내부 인증 헤더 인터셉터 패턴도 존재했다.
즉 팀 전체 코드 스타일과 일관성을 유지하기에도 좋았다.

따라서 이번 구현에서는:

- WebClient보다 FeignClient가 더 단순하고 설명 가능성이 높았다

라고 정리할 수 있다.

### 12-3. 실무에서 관리자 계정은 어떻게 관리하는가

정답은 하나가 아니고, 보통 서비스 성격과 보안 수준에 따라 달라진다.

가장 흔한 방식은 아래 세 가지다.

**1. 관리자 계정을 별도로 운영**

- 운영팀, CS팀, 관리자용 계정을 사전에 발급
- 일반 사용자 계정과 분리
- 가장 흔하고 안전한 방식 중 하나

장점:

- 일반 회원과 관리자 권한이 섞이지 않는다
- 권한 회수와 감사를 분리하기 쉽다

**2. 일반 계정을 관리자 권한으로 승격**

- 특정 사용자의 role을 `ADMIN`으로 올림
- 내부 운영 도구나 초기 세팅 단계에서는 간단하게 쓸 수 있다

장점:

- 구현이 쉽다

단점:

- 잘못 설계하면 권한 관리가 느슨해질 수 있다
- 운영 감사 관점에서 일반 계정과 관리자 계정이 섞이기 쉽다

**3. SSO / 사내 계정 연동**

- Google Workspace, Okta, Azure AD 같은 사내 계정 시스템과 연결
- 관리자 인증과 권한을 별도 IdP에서 관리

장점:

- 보안, 계정 회수, MFA 적용이 강하다

단점:

- 구현 및 운영 비용이 크다

이번 프로젝트처럼 포트폴리오/실습 수준에서는:

- `ADMIN` role을 둔 뒤 관리자 계정을 별도로 생성해 사용하는 방식

이 가장 무난하다.

즉 실무적으로도:

- "일반 회원을 승격시키는 API를 무조건 제공해야 한다"는 건 아니다
- 오히려 관리자 계정은 별도 발급하고, 필요 시 DB/운영툴/백오피스로 권한을 부여하는 쪽이 더 흔하다

### 12-4. 외부 API 장애 전파는 어떻게 대응해야 하나

항상 정답이 서킷 브레이커는 아니다.

먼저 봐야 할 것은 이 세 가지다.

1. 호출 중요도
2. 트래픽 규모
3. 실패 시 사용자 영향

이번 관리자 통합 조회는:

- 관리자용 조회
- 상대적으로 저트래픽
- 내부 서비스 한 곳이 죽으면 전체 응답을 구성하기 어려움

이런 특성이 있다.

그래서 현재 단계에서는:

- 실패 시 `503 SERVICE_UNAVAILABLE`
- 로그 기록
- 필요하면 timeout

정도로도 충분히 타당하다.

즉 초기 대응은 보통 이렇게 간다.

**1단계. timeout**

- 무한 대기 방지
- 가장 먼저 필요하다

**2단계. 예외 변환**

- 내부 호출 실패를 서비스 장애 의미로 정리
- 이번 구현의 `SERVICE_UNAVAILABLE`이 여기에 해당한다

**3단계. 재시도 여부 판단**

- idempotent 조회 호출이면 짧은 재시도를 고려할 수 있다
- 하지만 무분별한 재시도는 장애를 더 키울 수 있다

**4단계. circuit breaker**

- 장애가 반복되거나, 특정 서비스 지연이 전체 시스템에 영향을 줄 때 도입
- 호출 차단, 빠른 실패, fallback 전략이 필요할 때 효과적이다

즉 결론은:

- 모든 내부/외부 API 호출에 처음부터 서킷 브레이커를 넣는 건 과할 수 있다
- 현재 요구사항에 맞는 최소 대응은 timeout + 예외 정리 + 필요시 짧은 재시도다
- 트래픽과 장애 패턴이 커지면 그때 circuit breaker를 붙이는 것이 더 실무적이다

이번 관리자 조회 API에서는:

- 저트래픽 관리자 기능
- 단일 조회 API
- 장애 시 503 반환이 합리적

이므로, 서킷 브레이커를 당장 필수로 보진 않았다.
