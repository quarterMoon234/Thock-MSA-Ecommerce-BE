# Redis Lua Gate 재고 선차감 실험 분석

## 1. 실험 목적

이 실험의 목적은 재고 예약 요청이 폭주하는 상황에서:

- **Redis Lua gate 미적용**
- **Redis Lua gate 적용**

두 구조를 같은 조건에서 비교해, DB 비관적 락 경로 진입 수와 응답시간이 어떻게 달라지는지 검증하는 것이다.

핵심 질문은 두 가지였다.

- 품절이 예정된 경쟁 구간에서 모든 요청을 DB 비관적 락 경로까지 태워야 하는가
- Redis에서 선차감/선차단을 수행하면 DB 경합과 응답시간을 얼마나 줄일 수 있는가

즉 이 실험은 단순 캐시 실험이 아니라, **품절 직전 경쟁 구간에서 DB 락 경로를 얼마나 우회시킬 수 있는지**를 검증하는 구조다.

---

## 2. 문제 상황

기존 구조에서는 재고 예약 요청이 들어오면 모두 DB 비관적 락 경로로 들어갔다.

재고가 10개이고 예약 요청이 1,000건이면:

- 실제로 성공하는 예약은 10건
- 나머지 990건은 결국 실패

이지만,

- 실패할 요청도 일단 DB 락 경로에 진입
- 락 대기와 트랜잭션 비용을 지불
- 응답시간이 불필요하게 커짐

즉 병목의 본질은 “성공할 수 없는 요청도 DB까지 들어간다”는 점이다.

---

## 3. 구현 방향

Redis Lua gate는 예약 요청이 DB에 들어가기 전에 아래를 먼저 판별한다.

- 이미 Redis 기준 예약 가능 수량이 남아 있는가
- 아니면 선차감 시점에서 바로 품절로 차단해야 하는가

핵심 아이디어는 다음과 같다.

- **성공 가능성이 있는 요청만 DB 경로에 진입**
- **나머지 요청은 Redis에서 즉시 reject**

즉 DB는 “정말 처리할 가치가 있는 소수의 요청”만 받고, Redis가 대량 실패 요청을 앞단에서 잘라낸다.

---

## 4. 비교 대상

### BEFORE: Redis gate 미적용

- 모든 예약 요청이 DB 비관적 락 경로에 진입
- DB에서 성공/실패가 최종 결정됨

### AFTER: Redis gate 적용

- Redis Lua script가 선차감/선차단 수행
- 성공 가능한 요청만 DB 경로 진입
- 초과 요청은 Redis에서 즉시 reject

---

## 5. 테스트 조건

최종 수치에 사용한 run은 아래 3개다.

- `1775907298`
- `1775907482`
- `1775908488`

공통 조건:

- 예약 요청 수: `1,000건`
- 초기 재고: `10`
- 요청 수량: `1`
- executor: `per-vu-iterations`
- 가상 사용자 수: `1,000 VUs`
- 각 VU는 예약 요청 `1회` 수행
- `product-service`는 `experiment` 프로필로 실행

즉 **재고 10개에 대해 1,000개의 예약 요청을 거의 동시에 밀어 넣는 burst 경쟁 상황**을 재현한 실험이다.

## 6. 부하는 어떻게 주었는가

실행 스크립트는 아래다.

- `loadtest/run-product-stock-redis-experiment.sh`

실제 k6 스크립트는 아래다.

- `loadtest/product-stock-pessimistic-baseline.js`

흐름은 다음과 같다.

3. Redis gate OFF / ON 두 phase를 각각 실행

즉 이 실험의 입력은:

- **예약 요청 1,000건 동시 투입**

이고,

측정 대상은:

- 예약 응답시간
- Redis 선차단 수
- DB 경로 진입 수

이다.

## 7. 측정 기준

이번 실험에서 중요하게 본 값은 아래다.

- `stock_reservation_duration.avg`
  - 예약 요청 평균 응답시간
- `stock_reservation_duration.p95`
  - 예약 요청 p95 응답시간
- `stock_pressure_redis_pre_rejected_count`
  - Redis에서 DB 진입 전에 차단된 요청 수
- `stock_pressure_db_entry_count`
  - 실제 DB 비관적 락 경로에 진입한 요청 수
- `stock_pressure_db_succeeded_count`
  - 실제 예약 성공 수

여기서 `avg`, `p95`는 **사용자가 받는 예약 API의 실제 HTTP 응답시간** 기준이다.

즉 이 값에는 아래 구간이 포함된다.

- HTTP 요청 진입
- Redis Lua gate 실행
- 필요 시 DB 비관적 락 경로 처리
- 응답 생성 및 반환

따라서 이 실험의 응답시간 개선은 단순히 Redis script 실행 시간이 빨라졌다는 뜻이 아니라,
**실패할 요청을 DB에 보내지 않으면서 예약 API 전체 응답시간이 줄었다**는 의미다.

핵심 해석은 아래와 같다.

- Redis gate 미적용
  - DB 경로 진입 `1000`
  - 성공 `10`
  - 실패 `990`

- Redis gate 적용
  - Redis 선차단 `990`
  - DB 경로 진입 `10`
  - 성공 `10`

즉 “성공 수는 같지만, 실패 요청 990건을 어디서 걸러냈는가”를 보는 실험이다.

## 8. 최종 결과

기준:

- 3회 반복 부하 실험 중앙값
- 재고 `10`
- 예약 요청 `1,000건`

| 항목 | BEFORE (`Redis gate 미적용`) | AFTER (`Redis gate 적용`) | 개선율 |
|---|---:|---:|---:|
| 평균 응답시간 | 534.52ms | 225.05ms | 57.9% |
| p95 응답시간 | 938.81ms | 404.60ms | 56.9% |
| DB 경로 진입 수 | 1000 | 10 | 99.0% 감소 |
| Redis 선차단 수 | 0 | 990 | - |
| 실제 예약 성공 수 | 10 | 10 | - |

해석:

- 실제 성공 예약 수는 동일하게 `10건`
- 그러나 Redis gate 적용 후에는 실패할 `990건`을 DB까지 보내지 않음
- 그 결과 DB 락 경로 진입은 `1000 -> 10`
- 평균 응답시간과 p95 모두 유의미하게 감소

즉 이 실험은 **성공률을 바꾸지 않고, 실패 요청의 비용만 줄여 응답시간과 DB 경합을 낮췄다**는 점을 증명한다.

---

## 9. 왜 3회 중앙값을 썼는가

이 실험은 correctness 실험이 아니라 성능 실험이다.

즉 핵심은:

- 단일 run에서 우연히 빠른 값이 나왔는가
- 환경 변동을 감안해도 개선 경향이 유지되는가

이므로 중앙값 기준이 적절하다.

이번 실험에서는:

- BEFORE 3개 run
- AFTER 3개 run

의 중앙값을 사용해 이력서 수치를 고정했다.

---

## 10. 면접에서 설명할 때의 핵심 문장

짧게 설명하면 아래가 가장 정확하다.

> 재고 10개에 예약 요청 1,000건이 동시에 들어오면, 기존 구조는 실패할 요청 990건도 모두 DB 비관적 락 경로까지 들어갔습니다. Redis Lua gate를 두고 선차감/선차단을 적용해 성공 가능한 요청 10건만 DB로 보내도록 바꿨고, 그 결과 DB 경로 진입을 1,000건에서 10건으로 줄이면서 평균 응답시간 57.9%, p95 56.9% 개선을 확인했습니다.

조금 더 기술적으로 설명하면 이렇게 이어갈 수 있다.

> Redis Lua script가 재고와 예약 상태를 원자적으로 검사하고 선차감하므로, 품절이 예정된 대량 실패 요청은 DB에 들어가기 전에 바로 차단됩니다. 그래서 DB는 실제로 처리할 가치가 있는 소수 요청만 받고, 락 대기와 트랜잭션 비용이 크게 줄어듭니다.

---

## 11. 이 실험이 증명하는 것과 증명하지 않는 것

### 증명하는 것

- Redis Lua gate가 DB 비관적 락 경로 진입 수를 크게 줄인다는 점
- 품절 직전 경쟁 상황에서 응답시간을 낮춘다는 점
- 성공 수는 유지하면서 실패 요청 비용만 줄인다는 점

### 증명하지 않는 것

- 일반적인 상품 조회 성능
- 분산 락 전체 대체 가능성
- 모든 재고 시나리오에서의 최적값

이번 실험은 **품절 직전 burst 경쟁 구간에서 Redis gate의 효과**만 분리해서 본 것이다.
