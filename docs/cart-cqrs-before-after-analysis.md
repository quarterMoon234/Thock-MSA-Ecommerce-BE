# 장바구니 CQRS Before/After 부하 실험 분석

## 1. 실험 목적

이 실험의 목적은 장바구니 조회·생성 경로를 아래 두 구조로 비교하는 것이다.

- BEFORE: 상품 서비스 동기 호출 기반 경로
- AFTER: CQRS 읽기 모델 기반 경로

검증하고자 한 핵심은 두 가지다.

- 장바구니 조회·생성 응답시간이 실제로 줄어드는가
- 상품 서비스 지연이 장바구니 경로에 직접 전파되지 않게 되었는가

즉 이 실험은 단순 조회 속도 비교가 아니라, **CQRS 읽기 모델 분리로 동기 의존 제거와 응답 안정성이 함께 개선되는지**를 검증하는 구조다.

---

## 2. 문제 상황

기존 장바구니 경로는 상품 정보를 가져오기 위해 상품 서비스를 동기 호출했다.

이 구조의 문제는 다음과 같다.

- 장바구니 조회 시 상품 서비스 응답시간이 그대로 반영됨
- 장바구니 생성 시에도 상품 정보 확인 비용이 동기 호출로 들어감
- 상품 서비스 지연이나 장애가 장바구니 경로로 직접 전파될 수 있음

즉 성능 문제와 장애 전파 문제가 동시에 존재했다.

---

## 3. 구현 방향

장바구니 조회·생성에 필요한 상품 정보를 CQRS 읽기 모델로 분리했다.

핵심 아이디어는 다음과 같다.

- 조회·생성 경로는 읽기 모델을 직접 참조
- 정합성이 중요한 주문·결제 경로만 상품 서비스 동기 호출 유지

즉 사용자 트래픽이 많이 몰리는 장바구니 경로에서는:

- 상품 서비스 호출을 제거하고
- 읽기 모델로 필요한 데이터만 빠르게 조회

하도록 설계했다.

---

## 4. 비교 대상

### BEFORE: 상품 서비스 동기 호출

- 장바구니 조회 시 상품 서비스 호출
- 장바구니 생성 시 상품 서비스 호출
- 상품 서비스 지연이 장바구니 API 지연으로 직결

### AFTER: CQRS 읽기 모델 참조

- 장바구니 조회는 읽기 모델에서 상품 정보 조회
- 장바구니 생성도 읽기 모델을 기준으로 검증
- 상품 서비스 지연이 장바구니 API 경로에 직접 영향을 주지 않음

---

## 6. 20 VUs와 30초의 의미

### 20 VUs

`20 VUs`는 가상 사용자 20명이 동시에 요청을 보내는 뜻이다.

즉 각 시나리오에서:

- 요청 주체 20개가 동시에 동작하고
- 한 요청이 끝나면 다시 다음 요청을 보내며
- 그 상태를 지정한 시간 동안 유지한다

는 의미다.

### 30초

`30초`는 각 시나리오를 30초 동안 실행했다는 뜻이다.

즉 이번 실험은:

- “정확히 몇 번 요청한다”가 아니라
- **30초 동안 계속 요청을 보내면서 얼마나 처리되는지**

를 측정하는 duration 기반 실험이다.

그래서 총 요청 수는 고정값이 아니고, 응답시간에 따라 달라진다.

---

## 7. 조회 회원 수 50, 생성 회원 수 100의 의미

여기서 `조회 회원 수 50`은 **조회 요청을 50번 보냈다**는 뜻이 아니다.

정확히는:

- 조회 대상 회원을 50명 준비해두고
- 요청마다 그중 한 명의 장바구니를 조회한다

는 뜻이다.

마찬가지로 `생성 회원 수 100`도:

- 생성 요청 대상 회원을 100명 준비해두고
- 요청마다 그중 한 명에 대해 장바구니 추가를 수행한다

는 의미다.

이렇게 한 이유는:

- 같은 회원 1명만 반복 조회/생성하면 데이터가 지나치게 단순해지고
- 캐시/워밍 효과 때문에 결과가 치우칠 수 있기 때문이다.

즉 이 실험은 **조금 더 현실적인 read/write 분산**을 만들기 위해 여러 회원을 준비했다.

## 8. 부하는 어떻게 주었는가

실행 스크립트는 아래다.

- `loadtest/run-cart-cqrs-experiment.sh`

실제 k6 스크립트는 아래다.

- `loadtest/cart-cqrs-experiment.js`

흐름은 다음과 같다.

1. 실험용 상품 3개 생성
2. 장바구니 실험용 dataset seed
3. 8개 시나리오를 각각 `20 VUs`, `30초`로 실행
4. 각 시나리오별 요청 응답시간과 요청 수를 수집

즉 입력은:

- **동시 사용자 20명이 30초 동안 계속 조회/생성 요청 수행**

이고,

측정 대상은:

- 평균 응답시간
- p95 응답시간
- 성공률
- 총 요청 수

이다.

## 9. 전체 요청 수는 몇 회였는가

이번 실험은 duration 기반이라 총 요청 수는 run마다 조금씩 달랐다.

3회 run의 전체 요청 수는 아래와 같다.

- `1776172280`: `203,853회`
- `1776173204`: `206,996회`
- `1776173660`: `202,496회`

즉 전체적으로 약 `20만 건` 수준의 요청이 발생했다.

대표 run `1776172280`의 시나리오별 요청 수는 아래와 같다.

- `sync_read`: `20,722`
- `cqrs_read`: `48,432`
- `sync_read_delay`: `320`
- `cqrs_read_delay`: `49,711`
- `sync_add`: `18,738`
- `cqrs_add`: `32,330`
- `sync_add_delay`: `320`
- `cqrs_add_delay`: `33,280`

느린 시나리오는 같은 30초 안에 처리 가능한 요청 수가 적기 때문에, 요청 수가 적게 나온다.

---

## 10. avg와 p95는 무엇을 의미하는가

이번 표의 `avg`와 `p95`는 모두 **실제로 발생한 개별 요청 응답시간들**을 기준으로 계산한 값이다.

예를 들어:

- `장바구니 조회 avg 28.75ms`

는 30초 동안 `sync_read` 시나리오에서 발생한 모든 조회 요청 응답시간의 평균이다.

또:

- `장바구니 조회 p95 55.04ms`

는 해당 요청들 중 **95%가 55.04ms 이하로 끝났다**는 뜻이다.

즉:

- `avg`: 전체 요청 응답시간 평균
- `p95`: 느린 꼬리 구간까지 반영한 95 percentile

이다.

---

## 11. 측정 기준

이번 실험에서 핵심은 아래 세 가지다.

- 조회 avg / p95
- 생성 avg / p95
- 상품 서비스 지연 주입 시 avg

해석은 다음과 같다.

- 일반 경로:
  - CQRS 적용 전후 평균/지연시간 차이 확인
- delay 경로:
  - 상품 서비스가 느려졌을 때 장바구니 경로가 직접 영향받는지 확인

즉 이 실험은 **성능 개선 + 장애 전파 완화**를 같이 본다.

## 12. 최종 결과

기준:

- 3회 반복 부하 실험 중앙값

| 항목 | BEFORE | AFTER | 개선율 |
|---|---:|---:|---:|
| 장바구니 조회 avg | 28.75ms | 12.25ms | 58.5% |
| 장바구니 조회 p95 | 55.04ms | 18.08ms | 67.2% |
| 장바구니 생성 avg | 31.41ms | 17.66ms | 43.75% |
| 장바구니 생성 p95 | 42.03ms | 23.07ms | 45.1% |
| 상품 서비스 지연 주입 시 조회 avg | 1811.50ms | 12.12ms | 99.33% |
| 상품 서비스 지연 주입 시 생성 avg | 1812.70ms | 17.86ms | 99.01% |

해석:

- 조회·생성 모두 응답시간이 줄었다
- 상품 서비스 지연이 생겨도 CQRS 경로는 거의 직접 영향받지 않았다

즉 CQRS 읽기 모델 분리는:

- 단순한 micro-optimization이 아니라
- **동기 의존 제거에 따른 응답 안정성 개선**

까지 가져왔다.

---

## 13. 왜 중앙값을 썼는가

이번 실험은 correctness가 아니라 성능 실험이다.

즉 핵심은:

- 단일 run이 우연히 빠른가
- 반복해도 개선 경향이 유지되는가

이므로 중앙값 기준이 적절하다.

이번 실험에서는 3회 반복 결과의 중앙값을 사용해 이력서 수치를 고정했다.

---

## 14. 면접에서 설명할 때의 핵심 문장

짧게 설명하면 아래가 가장 정확하다.

> 장바구니 조회·생성 경로를 CQRS 읽기 모델로 분리해 상품 서비스 동기 의존을 제거했습니다. 20 VUs, 30초 조건의 3회 반복 부하 실험 중앙값 기준 조회 평균 응답시간은 58.5%, 생성 평균 응답시간은 43.75% 단축됐고, 상품 서비스에 900ms 지연을 주입해도 장바구니 경로가 직접 영향을 받지 않는 것을 확인했습니다.

조금 더 기술적으로 설명하면 이렇게 이어갈 수 있다.

> 기존에는 장바구니 조회·생성이 상품 서비스 호출에 직접 묶여 있어 지연이 전파됐습니다. CQRS 읽기 모델로 필요한 상품 정보를 로컬 read path에서 읽도록 바꿨고, 그 결과 일반 경로 응답시간뿐 아니라 dependency delay isolation도 함께 개선됐습니다.

---

## 15. 이 실험이 증명하는 것과 증명하지 않는 것

### 증명하는 것

- 장바구니 조회·생성 경로의 응답시간 개선
- 상품 서비스 동기 의존 제거 효과
- 상품 서비스 지연 전파 차단

### 증명하지 않는 것

- 전체 시스템 end-to-end 최종 사용자 latency
- 주문·결제 경로까지 포함한 정합성
- CQRS의 eventual consistency 지연 자체

이번 실험은 **장바구니 조회·생성 read/write path의 성능과 의존성**만 분리해 검증한 것이다.

## 16. 실행 방법

실행 스크립트:

- `loadtest/run-cart-cqrs-experiment.sh`

실행 예시:

```bash
K6_SCENARIO_VUS=20 \
K6_SCENARIO_DURATION=30s \
K6_SCENARIO_SLOT_SECONDS=35 \
K6_PRODUCT_COUNT=3 \
K6_READ_MEMBER_COUNT=50 \
K6_ADD_MEMBER_COUNT_PER_SCENARIO=100 \
K6_PRODUCT_DELAY_MS=900 \
bash loadtest/run-cart-cqrs-experiment.sh
```

결과 파일:

- `loadtest/results/cart-cqrs-experiment-<run_id>.json`

이 JSON 하나로 BEFORE/AFTER 표와 지연 주입 비교를 함께 만들 수 있다.
